# Approved Plan: Git Context Capture

## Objective

- Add bounded git-context capture to the lifecycle helpers so `begin` and
  `close` record useful branch/dirty/changed-file information for runs and
  verification artifacts when a Git repository is available, while degrading
  cleanly outside Git repositories.

## Acceptance Criteria

- `begin` records a lifecycle run artifact under `.sb-codex-tool/runs/` that
  includes the cycle identity, phase, and git context when available.
- `close` records a lifecycle run artifact under `.sb-codex-tool/runs/` that
  includes the cycle identity, phase, verdict, and git context when available.
- the generated handoff for a new cycle includes a readable git-context section
  with branch, dirty state, and changed files when available, or an explicit
  unavailable note otherwise.
- the verification summary generated by `close` includes the same git-context
  evidence for the closing run.
- `status` and the hot-path state artifacts expose the latest lifecycle run
  without forcing the verifier to guess where git-context evidence lives.
- tests cover both Git-available and Git-unavailable paths without requiring
  the main workspace itself to be a Git repository.

## Boundaries

- In scope:
  - reusable run-record helper logic
  - begin/close git-context capture
  - handoff and verification-summary git-context sections
  - regression tests using temporary Git repositories
- Out of scope:
  - commit creation
  - merge or rebase automation
  - deploy or release workflows
  - changing `status` beyond existing live git inspection
